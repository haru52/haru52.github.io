# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

Personal website for "haru" (<https://haru52.com/>), deployed to GitHub Pages. A static site built with Pug templates, SCSS, and a small TypeScript bundle compiled by webpack. There are no runtime tests.

## Common commands

- `make dev-install` — install npm deps and sync Vale styles (first-time setup).
- `npm run dev` — webpack-dev-server with auto-open; serves `docs/` with source maps.
- `npm run build` — production build into `docs/`. `prebuild` runs `npm run clean` first.
- `npm run clean` — remove `docs/` but restore `docs/CNAME` from git.
- `make lint` — full lint suite: npm scripts + Vale prose + yamllint + shellcheck + actionlint.
- `npm run lint` — runs every `lint:*` subscript (secretlint, markdownlint, prettier, markuplint for Pug, stylelint for SCSS, eslint). `lint:credentials` is skipped when `CI=true`.
- `npm run format` — auto-fix counterparts for markdown/prettier/stylelint/eslint.
- `npm run commit` (or `npm run cm`) — Commitizen-driven conventional commit.

## Architecture

### Build pipeline (webpack)

- `webpack.common.ts` is the shared config; `webpack.dev.ts` and `webpack.prod.ts` extend it via `webpack-merge`.
- Entry: `src/typescript/index.ts`. Output: `docs/`.
- HTML pages are generated by globbing `src/views/**/*.pug` (excluding `_*.pug` partials) and instantiating one `HtmlWebpackPlugin` per page in `initHtmlWebpackPlugins()`. To add a new page, drop a `*.pug` file under `src/views/` — no config change needed.
- SCSS → `docs/assets/stylesheets/main.css` via `MiniCssExtractPlugin`. Images and `.asc` files are emitted as `asset/resource` under `docs/assets/`.
- `pug-loader` is configured with `root: src/views`, so absolute Pug include paths resolve from there.

### Source layout

- `src/typescript/` — entry, `Application`, and feature classes wired through `tsyringe` DI (`@injectable`, `container.resolve(Application).run()`). When adding behavior, inject it into `Application` rather than calling from `index.ts`.
- `src/views/` — `layouts/_layout.pug` is the base; components in `components/_*.pug` (header/footer/GA); page templates at the top level extend the layout and override `content` / `breadcrumb_items` / `variables` blocks.
- `src/assets/stylesheets/` — `index.scss` is the entry, imports `custom.scss` and Bootstrap.
- Dependencies are Bootstrap 5, Font Awesome, Popper, tsyringe, reflect-metadata.

### Deployment flow

- `docs/` is committed to the repo (GitHub Pages serves from it; `docs/CNAME` pins the domain).
- `.github/workflows/test.yml` runs lint + build on push/PR.
- `.github/workflows/publish.yml` triggers on successful Test runs on `main`: runs `npm run build` and commits the regenerated `docs/` back to `main` via `EndBug/add-and-commit` (author: github_actions, message `build: publish website resources to deploy`). **Do not hand-edit files under `docs/` as a source of change** — they are regenerated from `src/` by CI.

## Conventions and tooling

- Node/npm versions are pinned via `engines` in `package.json` and `.node-version` / `.tool-versions`.
- Commit messages: Conventional Commits (`@commitlint/config-conventional`). Enforced by commitlint via the `commit-msg` husky hook and in CI. PR titles follow the same rule.
- Git flow: GitHub flow (feature branch → PR → merge to `main`).
- `lint-staged` (husky `pre-commit`) runs secretlint + prettier on every staged file, plus per-extension checks (markdownlint + Vale for md, markuplint for pug, stylelint for scss, eslint for ts/js, actionlint for workflows).
- ESLint extends `airbnb-base` + `airbnb-typescript/base` + `prettier`. Prettier formats everything; don't hand-format.
- Vale is the prose linter for the top-level `*.md` docs; run via `make lint-text` (requires `vale sync` from `make dev-install`).

## Things not to do

- Don't commit changes under `docs/` by hand — the Publish workflow overwrites it.
- Don't add `test` scripts claiming success; current `npm test` intentionally exits 1 and the test CI job is commented out in `test.yml` until real tests exist.
- Don't bypass commitlint/lint-staged hooks; CI re-runs them and will fail.
